# Phase 6.9 Hardening and Purchase E2E Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans. Steps use checkbox (`- [x]`) syntax for tracking.

**Goal:** Harden the Phase 6 commerce boundary and prove the customer purchase path and admin/customer principal separation through a real Playwright run against built NestJS, Next.js, and PostgreSQL services.

**Architecture:** Keep production Razorpay behavior unchanged. Add a narrowly gated `E2E_PAYMENT_MODE=stub` adapter path that can create deterministic test orders while still exercising the existing payment-signature verification and order-confirmation state transition. Add email-verification enforcement only at order placement, extract the CORS origin allowlist into a tested helper, and use Playwright against the real HTTP boundary for the customer flow and cross-principal checks.

**Tech Stack:** NestJS 11, Prisma 7, PostgreSQL 16, Next.js 16 App Router, Zod, Playwright, Jest, Vitest, Razorpay SDK wrapper.

## Global Constraints

- No raw SQL; use Prisma exclusively.
- Money stays integer paise in persistence and API responses.
- Production payment calls and HMAC verification must remain Razorpay-backed; the stub is available only when explicitly enabled for non-production E2E execution.
- Order placement must reject an authenticated customer whose `email_verified` value is false.
- CORS must allow only explicitly configured origins; unknown origins must be rejected.
- Every external API response touched by the work must be Zod-validated at the frontend boundary; no TypeScript cast may replace validation.
- Customer tokens must not authorize admin routes, and admin tokens must not authorize customer routes.
- Do not change or stage unrelated image-pipeline, Codex, or local environment files.
- Stop after verification and before `git commit`; the owner must approve the diff first.
