# GROWTH.md - Sunfabb Reach & Discoverability Plan

> **This is Phase 7 of `docs/PLAN.md`.** See the Phase 7 section there for the four-wave summary.
> This document is the detailed sub-plan for each wave and touchpoint.

Drafted 2026-07-07 from a live audit of sunfabb.com, the repo, and current (2026) channel research.
Companion to `docs/PLAN.md` (build plan).
This document covers how the site reaches people.

Guiding principle: **every touchpoint below is designed to run on automation after a one-time setup.**
Each sub-plan separates "build once" (code/config Claude can implement), "runs itself" (scheduled
automation), and "owner only" (accounts, KYC, budget, brand approvals - things only you can do).

## 1. Where the project stands today (audit, 2026-07-07)

### 1.1 What already works in our favor

- Live site at `https://sunfabb.com` (Vercel) with clean, crawlable URL structure
  (`/catalog`, `/catalog/[slug]`).
- Product detail pages are SSG + ISR (`revalidate = 30`, `generateStaticParams`) and serve full
  HTML server-side.
  This matters more than ever: most AI crawlers (GPTBot, ClaudeBot, PerplexityBot) do not execute
  JavaScript, so server-rendered HTML is a prerequisite for AI-search visibility.
  We already have it.
- All product images use `next/image` with an `alt_text ?? product.name` fallback chain.
- The data layer already has almost everything SEO needs: `Product.description`,
  `Product.care_instructions`, `ProductImage.alt_text`, `Category.description`, variant
  `price`/`stock_quantity`/`sku`, `hsn_code`, and DB `updated_at` timestamps.
- Cloudinary hosts all product images, which means OG images and social assets can be generated by
  URL transformation instead of manual design work.
- Small catalog (20-30 products) means feed quality, structured data, and per-product copy can be
  near-perfect with little effort.
- E-commerce backend (cart, checkout, Razorpay) is done through Phase 6.4, so the "money path" that
  ads and Shopping listings need is close.

## 2. Strategy: four waves, gated by readiness

Channels are sequenced so nothing is built before its prerequisites exist.
Sending traffic (or Googlebot) to a site that times out, lacks trust pages, or cannot take payment
wastes the effort.

| Wave | Theme | Gate to start | Status | Touchpoints |
|------|-------|---------------|--------|-------------|
| **Wave 0** | Be indexable | None | ✅ **Shipped 2026-07-17** (PR #26) | §3.1 technical SEO, §3.2 crawl performance, §3.5 Search Console/Bing, §3.13 analytics. |
| **Wave 1** | Be worth citing and following | Wave 0 shipped + owner provides business inputs | ⬜ todo | §3.3 trust pages + content engine, §3.4 AI/LLM search (GEO), §3.7 Business Profile, §3.8 social engine start |
| **Wave 2** | Be buyable everywhere | Phase 6 go-live (6.10): live payments, shipping, policies | ⬜ todo | §3.6 Google Shopping, §3.8 Meta catalog/product tagging, §3.9 WhatsApp Business, §3.10 ONDC, §3.11 email flows |
| **Wave 3** | Pay to amplify what converts | Wave 2 live + GA4 shows the funnel converts | ⬜ todo | §3.12 ads (Google PMax, Meta), marketplace expansion, review engine |

**Trust pages note (gap 6):** `/about`, `/contact`, `/privacy-policy`, `/terms`,
`/shipping-policy`, `/returns-policy`, `/faq` require owner business inputs (legal entity name,
GSTIN display preference, contact channels, return window, shipping coverage/timelines) before
Claude can draft the copy.
These pages are also prerequisites for Razorpay live mode and Google Merchant Center (Wave 2
gates), so their natural home is Wave 1 - finishing them at the same time as the content engine
and before commerce go-live.

Definition of done per wave is in each sub-plan's KPI block.

## 3. Touchpoint sub-plans

### 3.1 Technical SEO foundation (code, one PR)

**Goal:** make every crawler (Google, Bing, AI bots) able to discover, fetch, and understand every
product.

**Today:** bare `title`/`description` only; everything else absent (see gaps 2-4, 7-8).

**Build once (Claude implements, one `feature/seo-foundation` branch):**

1. `app/robots.ts` - allow all, explicitly allow AI search bots (see §3.4), disallow `/admin`,
   `/account`, `/cart`, `/checkout`, `/api`; point to sitemap.
2. `app/sitemap.ts` - home, `/catalog`, category-filtered catalog URLs, every product slug;
   `lastModified` from product `updated_at` (expose `updated_at` in the public product API +
   `lib/api.ts` zod schema first - small backend change).
3. `metadataBase`, `title.template` (`%s | Sunfabb`), site-wide `openGraph` + `twitter` defaults
   in `app/layout.tsx`.
4. Home page gets real metadata (keyword-bearing title like "Sunfabb - Premium Cotton Bedspreads,
   Towels & Table Linen Online India" + description).
5. Product pages: extend `generateMetadata` with `openGraph.images` (primary image via a Cloudinary
   transform sized 1200x630), price/availability, and `alternates.canonical`.
6. Catalog page: `generateMetadata` reflecting active category filter ("Bedspreads - Sunfabb
   Catalog"), canonical stripping filter permutations to the category-level URL.
7. `robots: { index: false }` on cart, checkout, all `/account/**` pages.
8. JSON-LD components (`components/seo/`):
   - `Organization` + `WebSite` (root layout, with `sameAs` to social profiles once they exist),
   - `Product` + `Offer` on product pages (price in INR from variants, availability from
     `stock_quantity`, image, brand "Sunfabb"),
   - `BreadcrumbList` on catalog + product pages (breadcrumbs already render visually, just
     unmarked),
   - `ItemList` on the catalog grid.
9. `app/manifest.ts`, `app/icon.png`, `apple-icon.png`, a real logo file in `public/`, root
   `opengraph-image` for non-product pages.
10. `next.config.ts`: `headers()` (basic security headers), redirect `www.sunfabb.com` -> apex if
    not already handled at Vercel level.
11. Footer upgrade: nav links to policy/about/contact pages (placeholders until Wave 1 ships the
    real pages), social icons (once handles exist), brand line.

**Runs itself:** sitemap and JSON-LD read live data, so they never need maintenance; ISR keeps them
fresh.

**Owner only:** nothing.
This wave needs zero accounts.

**Depends on:** nothing.
Branch off `main`, independent of the Phase 6 stack.

**KPI / done:** `robots.txt` + `sitemap.xml` return 200; Rich Results Test passes Product +
Breadcrumb on a product URL; a WhatsApp share of a product shows image + title + price context.

### 3.2 Crawl performance and infrastructure

**Goal:** every URL responds fast enough that crawlers and users never see the 24s cold start.

**Today:** home 0.13s (cached), `/catalog` 24.0s cold (SSR per-request against a sleeping Render
free-tier backend).
keep-alive.yml exists but is not effective (secret likely unset, and GitHub cron is best-effort).

**Build once:**

1. Verify/set the `BACKEND_URL` GitHub secret so keep-alive.yml actually pings (stopgap).
2. Convert `/catalog` from per-request SSR to ISR: read filters via a client boundary or move
   filtering client-side over an ISR-cached dataset; the existing web-vitals plan
   (`.omc/plans/2026-06-21-phase-web-vitals-optimization.md`) already scopes this - execute it.
3. Ensure `generateStaticParams` covers all products so no product page ever cold-SSRs.

**Owner only:** upgrade Render to Starter ($7/mo, no sleep) at Wave 2 go-live - already planned as
Phase 6.10; consider pulling it forward to Wave 0 since $7/mo buys crawlability immediately.

**KPI / done:** p95 TTFB under ~1.5s on `/catalog` and product pages measured cold; Search Console
crawl stats show no availability errors.

### 3.3 Content, on-page SEO, and trust pages

**Goal:** give Google and LLMs substantive text to rank and cite, and give customers trust signals.

**Today:** product/category descriptions exist as DB fields but are nullable and thin; no policy
pages; no editorial content; hero is a stock photo.

**Gate:** owner provides business inputs before Claude can draft trust-page copy:
- Legal entity name and address
- GSTIN (display in footer or not)
- Contact email + phone
- Return window and conditions
- Shipping coverage (states/cities) + estimated timelines

**Build once (Wave 1, after inputs arrive):**

1. Static trust pages (required for Merchant Center, Razorpay live, and ads): `/about`, `/contact`,
   `/privacy-policy`, `/terms`, `/shipping-policy`, `/returns-policy`, `/faq`.
   Implement as simple static routes in the `(storefront)` group; Claude drafts the copy from your
   business inputs.
2. A lightweight guides section: `app/(storefront)/guides/[slug]` backed by MDX files in the repo
   (no CMS needed at this scale).
3. Category page intro copy blocks: render `Category.description` on filtered catalog views (field
   already exists, currently unused on the page).
4. An "SEO completeness" nudge in the admin product form: show which products are missing
   `description`, `care_instructions`, or image `alt_text` so data quality is visible at entry time.

**Runs itself (the automation that saves you weeks):**

- Claude batch-generates first-pass copy for all 20-30 products (descriptions in brand voice, care
  instructions, image alt texts) and all categories, delivered as a reviewable batch you approve
  once; applied via the existing admin API.
- Claude drafts 2 guides/month from a keyword map (examples for this niche: "Towel GSM explained:
  how to pick", "How to wash and care for cotton bedspreads", "Bedspread size guide for Indian bed
  sizes", "Table linen setting guide").
  These are exactly the pages LLMs cite and Google features in snippets.
- Quarterly content refresh pass: re-run the keyword map against Search Console query data, update
  underperforming pages.

**Owner only:** approve brand voice once (Claude produces a one-page voice guide first); supply real
product photography to replace the Unsplash hero (a phone + daylight is fine; stock photos undermine
both brand and image SEO).

**Depends on:** §3.1 for the guide pages to be indexed properly.

**KPI / done:** all products have description + alt text; all 7 trust pages live and linked from
footer; 4+ guides live by end of Wave 1; Search Console shows impressions on non-brand queries.

### 3.4 AI / LLM search (GEO - generative engine optimization)

**Goal:** when someone asks ChatGPT/Claude/Perplexity/Gemini "best cotton bedspreads online India",
Sunfabb is fetchable, parseable, and citable.

**Today:** nothing configured; but SSR HTML (the hard part) already exists.

**How this channel actually works (2026):** AI assistants retrieve via dedicated crawlers that read
raw HTML without executing JS.
Retrieval/citation bots (OAI-SearchBot, ChatGPT-User, Claude-SearchBot, PerplexityBot, BingBot
which feeds Copilot) drive citations; training bots (GPTBot, ClaudeBot, Google-Extended,
Applebot-Extended, Amazonbot, Meta-ExternalAgent) drive long-term brand presence inside models.
For a small brand seeking reach, allow both groups - being in training data is free distribution,
and there is no meaningful scraping downside for a public catalog.

**Build once:**

1. In `app/robots.ts`, explicitly `Allow: /` for the bots above (and keep `/admin`, `/account`,
   `/cart`, `/checkout` disallowed for all).
2. `public/llms.txt`: a curated markdown map (brand one-liner, category links, top products,
   guides, policies).
   Low cost, adopted by many tech companies; honest caveat: impact is still unproven, so spend 30
   minutes on it, not more.
3. Semantic HTML pass: ensure product pages use one `h1`, clear headings, tables/lists for specs
   (size, material, GSM, care) - LLMs extract structured facts far better from lists/tables than
   prose.
4. FAQ content (§3.3) with question-shaped headings; JSON-LD `FAQPage` on `/faq`.
5. Consistent entity signals: identical brand name, description, and `sameAs` links across
   Organization schema, socials, and directories.

**Runs itself:**

- GA4 referral segment for AI sources (`chatgpt.com`, `perplexity.ai`, `claude.ai`,
  `gemini.google.com`, `copilot.microsoft.com`) - a saved report, zero ongoing work.
- Monthly automated spot-check: a scheduled Claude agent asks the major assistants 5 canonical
  queries ("buy cotton bedspreads online India", etc.) and logs whether/what they cite - a simple
  visibility tracker in `docs/` or `.omc/`.

**Owner only:** nothing.

**Depends on:** §3.1 (robots), §3.2 (bots won't wait 24s), §3.3 (content worth citing).

**KPI / done:** AI bots appear in Vercel logs fetching product pages; first AI-referral sessions in
GA4; brand cited for at least one query in the monthly spot-check.

### 3.5 Search Console, Bing, and indexing plumbing

**Goal:** submit the site to the two ecosystems that feed both classic search and AI assistants, and
get monitoring for free.

**Today:** no verification, no submission anywhere.

**Build once:**

1. Google Search Console: verify `sunfabb.com` (domain property via DNS TXT - owner adds one
   record; or Claude adds the `verification.google` token to metadata).
   Submit sitemap.
2. Bing Webmaster Tools: import from Search Console (one click) - this feeds Bing, Copilot,
   DuckDuckGo, and ChatGPT's search index.
3. IndexNow: add the key file + a tiny ping on product publish/update (Bing/Yandex support it;
   Google ignores it; near-zero cost since the admin API already knows when products change).

**Runs itself:** Search Console emails coverage/enhancement issues automatically; a weekly scheduled
report (§3.13) pulls clicks/impressions/queries via the Search Console API.

**Owner only:** one DNS TXT record (or approve the meta-tag route); create the two free accounts
(Google account you already have; Microsoft account if none).

**Depends on:** §3.1.

**KPI / done:** all products indexed (Coverage report); zero critical enhancement errors; weekly
query report flowing.

### 3.6 Google Shopping (Merchant Center free listings, then ads)

**Goal:** products appear in the Google Shopping tab, image search, and (Wave 3) Performance Max
ads - the single highest-intent surface for product queries.

**Today:** nothing.
Gated on commerce go-live (buyable landing pages are required).

**India-specific requirements (verified 2026):** prices must be GST-inclusive and match the landing
page exactly; shipping settings (or `shipping` attribute) are mandatory for free-listing eligibility
in India; return/refund policy and contact info pages must exist; brand + GTIN/MPN expected (for
house-brand textiles without GTINs: set `brand: Sunfabb`, `mpn` = variant SKU,
`identifier_exists: no` where applicable).

**Build once (the automation core):**

1. Backend feed endpoint `GET /feeds/google-merchant.xml` (or TSV): one item per **variant**
   (size/color/material are Shopping attributes), mapping `sku` -> `id`/`mpn`, name + variant ->
   `title`, description, product URL (+ variant params), Cloudinary image, `price` in INR
   GST-inclusive, `availability` from `stock_quantity`, `item_group_id` = product id,
   `google_product_category` = "Home & Garden > Linens & Bedding ...", `hsn_code` already exists
   if ever needed for tax mapping.
   Because it reads the live DB, the feed is permanently self-updating - no manual re-uploads ever.
2. Feed unit tests (same standard as every module).

**Runs itself:** Merchant Center scheduled fetch pulls the feed daily; stock/price changes propagate
automatically.

**Owner only:** create the Merchant Center account, verify the website, complete business info +
phone verification, configure shipping zones/rates and return policy in the MC dashboard, accept
free-listings terms.
(All ~1-2 hours one time; Claude prepares a click-by-click checklist.)

**Depends on:** Phase 6.10 go-live, §3.1 structured data (MC cross-checks JSON-LD against the
feed), §3.3 policy pages.

**KPI / done:** 100% of active variants approved in MC; free-listing clicks visible in the MC
performance report.

### 3.7 Google Maps / Google Business Profile (GBP)

**Goal:** show up on Maps and the local pack, and own the brand knowledge panel.

**Decision needed from you (this branches the plan):**

- **Branch A - there is a physical presence** (shop, showroom, or office where a customer could
  visit or pick up): full GBP is available: create profile, verify (postcard/video), set category
  ("Linens store" / "Home goods store"), hours, photos, link website, enable messaging.
  Products can be surfaced on the profile; posts can be automated monthly from the same content
  engine as §3.8.
  Reviews become the growth loop: the post-purchase email (§3.11) links to the GBP review URL.
- **Branch B - online-only, no customer-facing location:** GBP is not eligible (it requires
  in-person customer contact).
  Skip Maps; the entity work still happens via `Organization` JSON-LD, consistent NAP-less brand
  citations, and social profiles.
  Revisit if you later add a pickup location or exhibition presence.

**Owner only (Branch A):** account creation and verification are inherently owner-only (Google
verifies the real-world business).

**KPI / done (A):** verified profile; 10+ photos; first 5 reviews; profile appears for "sunfabb"
and category+city queries.

### 3.8 Social media engine (Instagram, Facebook, Pinterest, YouTube Shorts)

**Goal:** a self-running content pipeline that turns the product catalog into a steady social
presence, sized for one owner with no time.

**Today:** no accounts linked anywhere on the site; no social presence found.

**Channel picks for this niche (home textiles, India):**

- **Instagram** - primary: India's D2C discovery engine (229M+ users); Reels + product tags.
  Note: native in-app checkout is not available in India; tags link out to the site, which is
  exactly what we want.
- **Facebook** - same Meta pipeline, near-zero marginal cost; also unlocks Marketplace-style
  discovery and the Page needed for WhatsApp/ads.
- **Pinterest** - disproportionately strong for home decor; pins are evergreen search assets, not
  feed posts; Rich Pins come free once §3.1 OG metadata ships.
- **YouTube Shorts** - optional 4th; reuse Reels verbatim.

**Build once:**

1. Reserve `@sunfabb` handles everywhere (owner, 30 minutes, do this immediately even before
   posting - squatting risk is real).
2. Meta Business Suite: Page + Instagram professional account + **Commerce Manager catalog** fed by
   a second backend feed `GET /feeds/meta-catalog.csv` (same data as §3.6, Meta's CSV schema) -
   enables product tagging in every post and, later, catalog ads.
3. Pinterest business account + claim the website (meta tag) + catalog feed (Pinterest ingests the
   same feed format family).
4. Add social links to the site footer and `Organization.sameAs`.

**Runs itself (the core ask - no manual posting):**

- **Monthly content batch:** Claude generates a 12-16 post calendar per month (3-4/week) from the
  catalog: product spotlights, care tips (reusing §3.3 guides), styling ideas, festival/seasonal
  hooks (Diwali, wedding season, summer = towels).
  Each item = caption + hashtags + a Cloudinary transform URL that composes the product image into a
  branded 4:5/9:16 frame (text overlay, logo, palette) - no designer needed.
- **Scheduling:** push the batch through a scheduler (Metricool/Publer/Postiz free tiers cover
  IG+FB+Pinterest via one CSV bulk upload) - owner reviews the month's queue once, ~20 minutes/month.
  (Building direct Meta Graph API posting is possible later, but a scheduler is the pragmatic v1;
  Postiz is self-hostable if you want zero SaaS.)
- **UGC loop (Wave 3):** post-purchase email asks for a photo + tag; reposts fill the calendar for
  free.

**Owner only:** handle registration, Meta/Pinterest account creation, one monthly 20-minute review
of the queued content (or grant full auto-post once trust is established), occasional real
photos/videos of products (the single highest-leverage manual input - authentic beats generated in
this niche).

**Depends on:** §3.1 (OG cards make shares/Rich Pins work); catalog feed depends on §3.6's feed
work (shared code).

**KPI / done:** 12+ posts/month publishing without manual per-post work; profile -> site clicks in
GA4; product tags live on IG posts (Wave 2).

### 3.10 ONDC and marketplaces (India-specific reach)

**ONDC (Open Network for Digital Commerce):** government-backed network where you list once via a
Seller Network Participant app (e.g. Mystore, SellerApp, eSamudaay) and appear across 100+ buyer
apps (Paytm etc.), 400+ cities, without building anything.
Onboarding needs GST, business PAN, bank details; product upload via their dashboard (small catalog
= an afternoon).
Treat as a low-effort experiment in Wave 2; logistics can ride the same Shiprocket account planned
for Phase 6.6.

**Amazon/Flipkart:** deliberate *later* decision (Wave 3+): commissions compress margins and dilute
D2C data, but they are where a large share of textile buyers search.
If tried, start with 3-5 hero SKUs as a listing test rather than the full catalog.

**Owner only:** all account/KYC steps; Claude prepares product data exports matching each
platform's template so listing is copy-paste.

**KPI / done (ONDC):** live on one SNP; first ONDC order or a decision to drop it after 90 days.

### 3.12 Paid ads (deliberately last)

**Goal:** amplify a funnel that already converts; never buy traffic to an unproven site.

**Gate (all must be true):** payments live (6.10), trust pages live, GA4 purchase events verified
end-to-end, Merchant Center approved, and organic/social shows some baseline conversion.

**Sequence when gated open:**

1. **Google Performance Max for Shopping** first: it reuses the §3.6 feed, targets bottom-funnel
   product queries, and needs no creative production.
   Start ₹500-1,000/day, one campaign, 4-6 week learning window, target ROAS only after conversion
   volume exists.
2. **Meta catalog/retargeting** second: requires Meta Pixel + Conversions API (build once alongside
   GA4 events in §3.13); retarget product viewers and cart abandoners with catalog ads before any
   cold prospecting.
3. Cold prospecting (Meta interest audiences, Google demand gen) only after retargeting ROAS is
   proven.

**Build once:** GA4 + Pixel/CAPI event layer (§3.13), UTM conventions, a weekly spend/ROAS line
in the automated report.

**Owner only:** ad accounts, billing, budget ceiling decisions, approving creative (which the §3.8
engine generates).

**KPI / done:** blended ROAS target you set (typical D2C textile floor: 2.5-3x on retargeting
before scaling); kill-switch rule pre-agreed (e.g. pause anything under 1x after 14 days).

### 3.13 Analytics and measurement backbone (underpins everything)

**Goal:** one place that says which touchpoint is working, updated without manual effort.

**Today:** Vercel Speed Insights only (performance RUM, not analytics).

**Build once:**

1. GA4 via `@next/third-parties` (env-var measurement ID), with e-commerce events:
   `view_item_list`, `view_item`, `add_to_cart`, `begin_checkout`, `purchase` (fired on the
   order-confirmation state after `/payments/verify` succeeds).
2. Meta Pixel + Conversions API stub behind the same consent/config layer (dormant until §3.12).
3. A minimal consent notice consistent with the privacy policy (India DPDP-aware; simple banner
   suffices at this scale).
4. UTM conventions doc + templated links used by the §3.8 scheduler and §3.11 emails.
5. **Weekly automated report:** a scheduled job (GitHub Action or a Claude scheduled agent) pulling
   Search Console API + GA4 Data API into a dated markdown report (`.omc/reports/` or emailed):
   impressions, clicks, top queries, sessions by channel, AI-referral sessions, conversions, and
   the §3.4 AI-citation spot-check monthly.

**Owner only:** create the GA4 property (10 minutes); grant API access for the report job.

**KPI / done:** purchase event visible in GA4 DebugView on a test order; first weekly report
generated without human touch.
