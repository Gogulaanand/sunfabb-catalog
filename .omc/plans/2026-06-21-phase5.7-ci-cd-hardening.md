# Phase 5.7 — CI/CD & Quality Hardening

Status: **approved — decisions confirmed 2026-06-21**
Mode: Direct (detailed request; facts gathered via direct repo inspection + Render/Vercel/GitHub API calls, not interview)

## 0. Why this phase exists

Phase 6 (cart/checkout/Razorpay) is the next feature phase, and it's the first time real money and a
multi-step user flow enter this codebase. Per `CLAUDE.md`'s "single admin, no timeline pressure"
philosophy, this phase is the safety net built *before* that risk arrives: working CI, a real test
gate, and confidence that "merge to main" actually means "live in production" — not an assumption.

## 1. Findings (the facts this plan is built on)

These were verified directly against the live Render/Vercel/GitHub state, not inferred from docs —
several contradict what `HANDOFF.md`/PR descriptions assumed.

### 1.1 CI (`.github/workflows/ci.yml`) does not run tests at all
Confirmed by reading the file: `backend` and `frontend` jobs both run `lint` + `tsc --noEmit` only.
**No `npm test` step exists in CI for either app**, despite the backend having 16 spec files / ~72
unit tests and the project requiring "every module ships with tests" (CLAUDE.md rule 6). Tests have
only ever been run manually by whoever/whatever wrote the PR. This is the single biggest gap — fixing
it is Step 1, ahead of everything else.

### 1.2 `main` has zero branch protection
`gh api repos/.../branches/main/protection` → 404 "Branch not protected". Nothing currently stops a
direct push to `main` or a merge with failing/no CI checks. CLAUDE.md rule: "main is always green...
branch protection enforces this" — it doesn't, today.

### 1.3 Render auto-deploy is correctly configured and working
`sunfabb-backend` (`srv-d8rkpcnlk1mc73c18q7g`): `autoDeploy: "yes"`, `branch: "main"`,
`autoDeployTrigger: "commit"`, repo `Gogulaanand/sunfabb-catalog`. Deploy history confirms it fires
automatically: deploy `dep-d8rlbi6rnols73fiunpg` started 68 seconds after commit `20461c6` landed on
`main`. **No action needed here beyond the keep-alive secret (1.4).**

### 1.4 Keep-alive workflow is live but inert — `BACKEND_URL` secret was never set
`.github/workflows/keep-alive.yml` is merged to `main` (PR #6) and runs every 10 minutes, but
`gh secret list` returns zero secrets on this repo. The workflow's own no-op guard means it's been
silently doing nothing since merge — Render free-tier cold starts are unmitigated. Backend URL,
confirmed via Render API: `https://sunfabb-backend.onrender.com`.

### 1.5 Vercel auto-deploy is **not** wired to GitHub at all — this is a real gap, not an assumption
This is the headline finding. `sunfabb-storefront` (`prj_FGbfoWb9V3Td7Uzag79QaI6pVUKG`) shows
`live: false`, and its only two production deployments were both triggered by a Claude-Code-agent
actor running CLI `vercel deploy` from `feature/phase5-deploy` — **not** by a GitHub push webhook,
and **not** from `main`. There is currently no live connection between "merge PR to main" and "new
version on sunfabb.com" on the frontend side. Every deploy to date has been a manual CLI action.
This must be fixed (link the Vercel project to the GitHub repo, set Production Branch = `main`) before
any of the CI-gating work in Step 3 is meaningful for the frontend.

### 1.6 Neither platform natively gates deploy on GitHub Actions status
This is true even after 1.5 is fixed. Render's and Vercel's native Git integrations deploy on
push/webhook receipt — they do not wait for a separate GitHub Actions run to report success. Today
this is moot for Vercel (nothing is wired) and harmless for Render only because CI currently checks
nothing meaningful (1.1). Once 1.1 (real tests) and 1.2 (branch protection) are fixed, "merge blocked
until CI green" becomes the actual deploy gate in practice, since nothing reaches `main` without
passing CI first. **Decision needed (1.6.D):** is merge-gating sufficient, or do we want belt-and-
suspenders deploy-time gating too? See Step 3.

### 1.7 Frontend has no test runner at all
`frontend/package.json` has no `test` script, no Jest/Vitest, no `@testing-library/*`. Backend has
real Jest unit + e2e infra (`backend/jest` config in `package.json`, `backend/test/app.e2e-spec.ts`)
but **no coverage threshold configured** (`collectCoverageFrom` is set; `coverageThreshold` is not).

### 1.8 No e2e or visual regression suite exists in the repo
Prior QA passes (Phase 5.5, 5.5.1) used Playwright/chrome-devtools MCP tools interactively in-session;
nothing was checked in. `find -iname "*playwright*"` across the repo returns nothing.

### 1.9 Repo is public
`gh repo view` → `visibility: PUBLIC`. This matters for Step 4/6 cost: GitHub Actions minutes are
unlimited (not metered) for public repos, and Codecov's free tier covers public repos fully. No paid
tooling is required for this phase — flagged explicitly per your instruction, see Step 7.

## 2. Scope boundaries (what this phase deliberately does NOT do)

- No checkout/payment code (Phase 6).
- No 100% coverage mandate — thresholds are risk-proportional (business logic in services > DTOs/
  controllers > presentational React).
- No paid SaaS by default (Codecov free tier is free for this public repo, but is still optional —
  see Step 7 decision).
- No multi-environment (staging/prod) deploy pipeline — this is a single-admin, ~100-product catalog;
  one production environment is correctly scoped to project size.
- No Docker/containerization — out of scope, Render/Vercel native builds are sufficient at this scale.

## 3. Implementation steps

### Step 1 — CI actually runs tests (highest priority — closes 1.1)
- `backend` job in `ci.yml`: added `npm run test:cov` (unit, with coverage) after type-check, a
  Postgres service container + `npx prisma migrate deploy` as a from-scratch migration sanity check,
  and `npm run test:e2e -- test/compression.e2e-spec.ts` (the one e2e spec that doesn't need a real
  Prisma client). **Found and worked around a real blocker along the way (now D28 in
  `docs/DECISIONS.md`):** Prisma 7 has no native binary query engine — its wasm query compiler needs
  a runtime `import()` that Jest's CJS transform can't do without a full ESM migration. So
  `test/app.e2e-spec.ts` (boots the real `AppModule`/`PrismaService`) can't run under Jest at all
  right now; DB-backed e2e coverage comes from Step 6's Playwright suite instead, which drives the
  real built server over HTTP. Also fixed a smaller pre-existing bug: `test/jest-e2e.json` was
  missing the `.js`-extension moduleNameMapper the unit config has, so even the trivial
  `app.e2e-spec.ts` couldn't resolve its imports — fixed without copying over the unit config's
  Prisma-client mock (e2e specs that do need a real client should hit the real one).
- `frontend` job: blocked on Step 4 (no test runner exists yet). Once added, wire `npm test` in here
  too.
- New branch protection rule on `main` (Step 2) makes both required checks.

### Step 2 — Branch protection on `main` (closes 1.2)
- `gh api repos/.../branches/main/protection` PUT: require status checks (`Backend`, `Frontend` job
  names from `ci.yml`) to pass before merge, require PR review is optional (single-admin repo, skip),
  disallow direct pushes to `main` (or allow admin bypass only — your call, will ask before applying).
- This is the action that actually makes "main is always green" (CLAUDE.md) true instead of aspirational.

### Step 3 — Fix Vercel deploy wiring + decide on deploy-gating strategy (closes 1.5, 1.6.D)
- Link `sunfabb-storefront` to the GitHub repo with Production Branch = `main` (this has to happen in
  the Vercel dashboard — the Vercel API doesn't expose Git-link configuration for write access).
- Once linked, every PR also gets a Vercel preview deployment for free — useful for visually
  reviewing frontend changes before merge.
- **Decision 1.6.D, recommended default:** rely on merge-gating only (Step 1+2) — once `main` requires
  green CI to merge, both Render and Vercel auto-deploying immediately on every `main` push is correct
  and sufficient; this is also exactly the pattern `HANDOFF.md`'s own Render-decision rationale
  describes. Skip Vercel's "Wait for CI" toggle (Pro-plan feature, adds latency, redundant once merge
  is gated) unless you want the extra belt-and-suspenders. Will confirm before applying.
- **Status: still open — needs your manual action in the Vercel dashboard.** Not something I can do
  via API.
- **New finding while validating Steps 1+4/6/7 via real `main` merges this session:** Render's
  auto-deploy, which I'd confirmed working earlier in this same session (deploy fired ~1 min after
  PR #7's merge), did **not** fire for either of today's two `main` merges (PR #10 at 08:01 UTC, PR
  #11 at 08:38 UTC) — `mcp__render__list_deploys` still shows the PR #7 deploy as the latest one, no
  newer entries. `gh api repos/.../hooks` returns empty, consistent with Render using a GitHub App
  install rather than a classic repo webhook, so delivery logs aren't inspectable from here. This
  needs a look at the Render dashboard's Events/deploy log for `sunfabb-backend` directly — possibly
  a stuck/failed webhook delivery, a GitHub App permission issue, or something else not visible via
  the API surface available to me. Flagging rather than guessing further.

### Step 4 — Frontend test runner + coverage gates, both apps (closes 1.7)
- **Backend**: add `coverageThreshold` to the existing Jest config in `backend/package.json`.
  Proposed split — `services/**`(business logic): 80%, global floor: 60%. Wire `npm run test:cov`
  into CI (Step 1) with `--coverageReporters=json-summary` for the badge in Step 5.
- **Frontend**: add Vitest + React Testing Library (lighter than Jest+jsdom for a Next.js/Turbopack
  project, faster in CI). Initial real coverage target is necessarily small — there are currently zero
  frontend tests. Write tests for the things with actual logic: `lib/api.ts` (data fetching/shaping),
  `CatalogFilters`/`CatalogTransitionContext` (the most stateful client components), the admin
  CRUD forms' validation behavior. Skip testing pure-presentational server components. Threshold:
  start low (e.g. 40% global) and ratchet up over time rather than gating on a number that doesn't
  reflect real risk yet — flagging this as a judgment call, will confirm the number with you.

### Step 5 — Release/status visibility (closes item 3 of your request)
- **README badges**: CI status (`workflows/ci.yml/badge.svg`), and a manually-added "deployed via
  Render/Vercel" badge linking to the live URLs (no native "deploy status" badge API from either
  platform without a paid plan — use a static badge + the live links instead).
- **GitHub Environments**: create `production` environment, attach deployment URLs
  (`https://sunfabb.com`, `https://sunfabb-backend.onrender.com`) so the repo's "Environments" tab
  shows live status — this is free and native, no extra workflow needed since Render/Vercel both
  already report deployment status to GitHub via their native integrations once 1.5 is fixed.
- **Release notes**: adopt Conventional Commits for PR titles going forward (this repo already
  informally does this — `feat:`, `fix:`, `docs:` prefixes are already the convention per recent
  commit history) + GitHub's native auto-generated release notes (`gh release create vX.Y.Z
  --generate-notes`) on meaningful milestones (e.g. tag `v0.6.0` for end of this phase). No new
  tooling required — this is the "don't over-build a release train" option you asked for.

### Step 6 — Playwright e2e suite, checked into the repo (closes 1.8, item 5)
- New top-level `e2e/` directory, `@playwright/test` as a root-level devDependency (monorepo has no
  root `package.json` today — will add a minimal one scoped to housing `e2e/` + Playwright config).
- Golden path spec: home → catalog (filter) → product detail → admin login → admin CRUD (create
  category → product → variant → soft-delete variant), mirroring exactly the manual checklist already
  in `docs/PRE_PHASE5_VERIFICATION.md` and the Phase 5.5/5.5.1 ad-hoc QA passes.
- CI job: spin up a `postgres` service container, run `prisma migrate deploy` + a minimal seed
  (1 category/material/color, 1 admin user with a known test password), start backend (`start:dev`-
  equivalent built artifact) and frontend (`next build && next start`) as background processes, run
  Playwright against `localhost`, tear down. This is the same pattern Step 1's backend e2e job uses
  for Postgres, extended to cover both apps running together.

### Step 7 — Visual regression snapshots (closes item 6 of your request)
- Add `toHaveScreenshot()` specs for: home, catalog, product detail (storefront) and admin login +
  one CRUD list view (admin), each at desktop (1280×800) and the 390×844 mobile breakpoint already
  used as precedent in D23.
- **Must run only in CI** (`ubuntu-latest`, pinned Playwright Docker image) — local-machine screenshot
  diffs are not reliable across OSes/font rendering; baseline snapshots get committed from a CI run,
  not a local one.
- Run as a separate, non-blocking CI job initially (`continue-on-error: true` or a separate workflow)
  for the first few cycles while baselines stabilize, then promote to a required check once proven
  stable — visual diffs are more failure-prone than functional tests and shouldn't block merges until
  trusted.

## 4. Cost flags (per your instruction — nothing here is silently adopted)

- **Codecov**: optional. Free for this public repo. Value: PR coverage diff comments instead of just
  a pass/fail threshold. Not required — a plain `coverageThreshold` failure in CI step output gives
  the same gating without a third-party integration. Recommend skipping unless you want the PR-comment
  UX; zero cost either way given public repo.
- **GitHub Actions minutes**: unmetered for public repos — Playwright + visual regression + the new
  Postgres-service e2e job add real runtime (probably +3-5 min/run) but **zero billing impact**.
- **Vercel/Render plans**: no plan change needed for anything in this phase. Linking Vercel's Git
  integration (Step 3) and GitHub Environments (Step 5) are both free-tier features.
- **No other paid service is introduced anywhere in this plan.**

## 5. Sequencing & dependencies

Steps 1→2 must land first (they're the actual safety net) before Step 3 (Vercel fix) matters, since
gating only means something once `main` requires green CI. Step 4 (coverage) extends Step 1's new test
steps. Steps 6→7 (e2e, visual regression) depend on Step 4's frontend test infra existing for shared
tooling conventions but are otherwise independent and can be built in parallel once Step 1 lands.
Recommended PR breakdown: one PR per step (7 PRs), each independently revertable — consistent with
this repo's existing "small, phase-scoped PR" pattern.

## 6. Decisions — confirmed 2026-06-21

1. **1.6.D** — merge-gating only. No deploy-time CI-wait toggle on Vercel.
2. **Step 2** — branch protection allows admin bypass (required checks enforced for normal PRs; owner
   can bypass in an emergency).
3. **Step 4** — frontend coverage starting threshold: 40% global, ratchet up over time.
4. **Step 4** — Vitest for frontend tests.
5. **Step 7** — visual regression runs non-blocking initially; promote to required once baselines
   prove stable across a few cycles.
